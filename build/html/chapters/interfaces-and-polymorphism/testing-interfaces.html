8.5. Unit Testing and Interfaces

After all that we have learned about interfaces, perhaps you are wondering, how do I write my unit tests with interfaces?

The best practices to testing interfaces are very similar to those of testing inheritance. You want to focus on testing the contract that the interface is supposed to be upholding as opposed to the interface itself.

Example

We have an IFeedable interface and a HouseCat class.

 1public interface IFeedable
 2{
 3   string Eat()
 4   {
 5      return "the feedable is eating";
 6   }
 7}
 8
 9public class HouseCat : IFeedable
10{
11   public string Name { get; set; }
12   public HouseCat(string name)
13   {
14      Name = name;
15   }
16}

Based on the class above and the interface it implements, we should expect that a call to .Eat() on an instance of a HouseCat will return the default implementation.

 1[TestClass]
 2public class CatTests
 3{
 4   [TestMethod]
 5   public void TestHouseCatImplementsEatMethod()
 6   {
 7      IFeedable test_cat = new HouseCat("test");
 8      Assert.AreEqual("the feedable is eating", test_cat.Eat());
 9      // This test passes. Don't believe us? Try it yourself!
10   }
11}

In this situation, we test the contract that the interface is supposed to be upholding, but not the interface itself. While there are strategies to test interface code itself using mock objects, we won’t approach those tactics in this course.

8.5.1. Check Your Understanding

Question

True or False: A class PetDog also implements IFeedable and contains its own Eat() method. Should we test that method’s results?

  1. No, the interface contract will not be tested in this scenario.

  2. Yes, testing the class’s custom methods is always a worthy endeavor.
